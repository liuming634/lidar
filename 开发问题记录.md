## 1. libusb 符号冲突

### 错误信息
```
libpcl_io.so: undefined symbol: libusb_set_option
```

### 原因
- 海康相机 SDK（MVS）安装在 `/opt/MVS/`，自带了旧版 `libusb-1.0.so.0`
- `LD_LIBRARY_PATH` 中 `/opt/MVS/lib/64` 排在系统库路径前面
- 导致 PCL 加载时优先找到 MVS 自带的 libusb，缺少 `libusb_set_option` 符号

### 解决
在 `~/.profile` 中，将系统 libusb 路径放在 MVS 路径之前：

```bash
# 修改前
export LD_LIBRARY_PATH=/opt/MVS/lib/64:/opt/MVS/lib/32:$LD_LIBRARY_PATH

# 修改后
export LD_LIBRARY_PATH=/usr/lib/x86_64-linux-gnu:/opt/MVS/lib/64:/opt/MVS/lib/32:$LD_LIBRARY_PATH
```

系统 libusb 版本 1.0.25，兼容 MVS SDK。

## 2. cv::FileStorage 读取 YAML 崩溃

### 错误信息
```
OpenCV(4.5.4) ... Input file is invalid in function 'open'
Component constructor threw an exception: ...
```

### 原因
OpenCV 的 `cv::FileStorage` 在无法打开文件时会抛出 `cv::Exception`，节点构造函数中未捕获异常导致整个节点崩溃。

### 解决
1. **去掉 YAML 行内注释**：`cv::FileStorage` 的 YAML 解析器不支持行内注释（如 `field_width: 28.0 # 注释`），会解析失败。注释必须单独成行。
2. **添加 try-catch 保护**：所有 `cv::FileStorage` 读取操作包在 try-catch 中，失败时使用默认值。

```cpp
try {
    cv::FileStorage fs("./config/params/field_params.yaml", cv::FileStorage::READ);
    if (fs.isOpened()) {
        fs["field_width"] >> field_width_;
        fs["field_height"] >> field_height_;
        fs.release();
    }
} catch (const cv::Exception& e) {
    RCLCPP_WARN(this->get_logger(), "Failed to read config, use defaults: %s", e.what());
}
```

### 涉及节点
- `localization` → `config/params/gicp_crop.yaml`
- `dynamic_cloud` → `config/params/dynamic_crop.yaml`
- `kalman_filter` → `config/params/field_params.yaml`
- `resolve` → `config/params/field_params.yaml`
- `debug_map` → `config/params/field_params.yaml`

## 3. 构建设置提醒

### symlink-install
```bash
colcon build --symlink-install --cmake-args -DCMAKE_BUILD_TYPE=Release -DCMAKE_EXPORT_COMPILE_COMMANDS=1
```

使用 `--symlink-install` 可以让 install 目录中的文件链接回源文件，修改源码后不需要重新 build 即可测试（仅头文件修改或 Python 节点有效，C++ 仍需要编译）。

## 5. 不同海康相机型号适配问题

### 错误信息
```
[hik_camera_node] [INFO]: 找到 1 个相机
[hik_camera_node] [ERROR]: 打开设备失败
```

### 现象
换了一台海康相机后，原来的代码无法打开设备（`MV_CC_OpenDevice` 返回 `MV_E_ACCESS_DENIED`），但同型号旧相机正常。设备用 MVS 客户端可以正常打开。

### 原因
海康相机 SDK（MVS）对所有型号统一使用 `libMvCameraControl.so`，但不同型号依赖不同的传输层动态库。相机类型切换后：

1. **传输层依赖不同**：MV-CS016-10UC 是 USB3 Vision (U3V) 相机，依赖 `libMvUsb3vTL.so`。如果旧相机用的是其他传输层（如 GigE 的 `libMvGigEVisionTL.so`），代码里链接的是同一个 `libMvCameraControl.so`，但运行时加载的传输层库不同
2. **默认像素格式不同**：不同型号出厂默认像素格式不同（有的默认 `BGR8`，有的默认 `BayerRG8` 或 `RGB8`）。原代码对非 Bayer 格式直接裸拷贝后标注为 `bgr8`，如果新相机默认输出 `RGB8` 就会出现颜色错乱
3. **设备初始化时序**：新相机第一次上电时可能需要完整读取出厂参数（GenICam XML），如果初始化未完成或 `libusb` 加载版本不对，`MV_CC_OpenDevice` 会失败。连 MVS 打开过一次后设备状态恢复正常

### 解决

1. **明确设置像素格式**，不再依赖相机默认值：
   ```cpp
   MV_CC_SetEnumValue(handle_, "PixelFormat", PixelType_Gvsp_BGR8_Packed);
   ```

2. **按 Bayer 模式分别转换**，不再全部用 `BayerRG2BGR`：
   ```cpp
   if (frame.stFrameInfo.enPixelType == PixelType_Gvsp_BayerRG8)
       cv::cvtColor(bayer, bgr, cv::COLOR_BayerRG2BGR);
   else if (frame.stFrameInfo.enPixelType == PixelType_Gvsp_BayerGB8)
       cv::cvtColor(bayer, bgr, cv::COLOR_BayerGB2BGR);
   // ...
   else if (frame.stFrameInfo.enPixelType == PixelType_Gvsp_RGB8_Packed)
       cv::cvtColor(rgb, bgr, cv::COLOR_RGB2BGR);
   ```

3. **打印像素格式**用于调试：
   ```
   [hik_camera_node]: Pixel format: BGR8_Packed
   ```

4. **添加自动白平衡**，使颜色接近 MVS 客户端效果：
   ```cpp
   MV_CC_SetEnumValue(handle_, "BalanceWhiteAuto", 2);  // 连续自动白平衡
   ```

### 涉及文件
- `src/hik_camera/src/hik_camera_node.cpp` — 相机初始化参数设置 + 像素格式转换逻辑

### 如何排查新相机
1. 先跑 MVS 客户端确认相机能工作
2. 关掉 MVS，用测试程序确认 SDK 能打开
3. 编译运行后看 `Pixel format:` 日志，确认实际输出格式
4. 如有颜色问题，查 MVS 里的当前参数值（白平衡、曝光、增益）加到代码中

## 4. 海康相机 USB 权限

### 错误信息
```
[hik_camera_node] [ERROR]: 打开设备失败
```

### 原因
Linux 默认 `/dev/bus/usb/*/*` 只有 root 用户可访问，普通用户没有读写 USB 设备的权限。

### 解决

**方法一（推荐）：root 启动**
```bash
sudo env PATH=$PATH ros2 launch tdt_vision radar.launch.py
# sudo 默认会重置 PATH，需要显式传递才能找到 ros2 命令
```

**方法二：加入 dialout 组（推荐）**
USB 设备通常属于 `dialout` 组，加入后普通用户即可访问：
```bash
sudo usermod -aG dialout $USER
# 重新登录后生效，之后直接 ros2 launch ... 即可，不需要 sudo
```

**方法三：添加 USB 权限规则（持久化）**
```bash
echo 'SUBSYSTEM=="usb", MODE="0666"' | sudo tee /etc/udev/rules.d/99-usb-permissions.rules
sudo udevadm control --reload-rules
# 重新插拔相机后生效
```

方法二对系统更安全（不需要一直用 root）。方法一最简单。
